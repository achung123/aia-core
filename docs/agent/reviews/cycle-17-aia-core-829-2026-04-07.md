## Findings

### [MEDIUM] `dispose()` does not explicitly remove the dismiss button's click listener

**File:** `frontend/src/components/resultOverlay.js`
**Line(s):** 158–160
**Category:** design (resource management)

**Problem:**
`dispose()` calls `overlay.remove()` to detach the overlay from the DOM, but it does not call `removeEventListener` on the dismiss button nor null out the captured `overlay` variable in the closure. Each `show()` invocation creates a fresh `dismissBtn` and binds a new click handler that captures the outer `overlay` reference. After `dispose()`, the following reference chain keeps the overlay subtree alive for as long as the caller holds the returned `{ show, hide, dispose }` object:

```
returned API object → show/hide/dispose closures → overlay (detached) → dismissBtn (last) → click listener → overlay
```

Modern garbage collectors handle self-contained cycles, so this will not grow unboundedly in practice. However, the documented contract for `dispose()` — "clean up and release resources" — is not fully honoured: the active dismiss button's listener is never explicitly unregistered, and `overlay` is never nulled out in the closure. Any long-lived caller that stores the API object after calling `dispose()` will retain the entire subtree.

**Code:**
```js
function dispose() {
    overlay.remove();   // removes from DOM but closures still hold overlay alive
}
```

**Suggested Fix:**
Track the active dismiss button and remove its listener on dispose; optionally null out the `overlay` reference via a flag:

```js
let activeDismissBtn = null;
let activeDismissHandler = null;

// inside show():
activeDismissHandler = () => { overlay.style.display = 'none'; };
dismissBtn.addEventListener('click', activeDismissHandler);
activeDismissBtn = dismissBtn;

// dispose():
function dispose() {
    if (activeDismissBtn && activeDismissHandler) {
        activeDismissBtn.removeEventListener('click', activeDismissHandler);
        activeDismissBtn = null;
        activeDismissHandler = null;
    }
    overlay.remove();
}
```

**Impact:** Low risk in short-lived pages; becomes a latent memory issue in SPA contexts where overlays are created and destroyed repeatedly without page reloads.

### [LOW] Winner detection hard-codes `result === 'win'` with no contract comment

**File:** `frontend/src/components/resultOverlay.js`
**Line(s):** 138
**Category:** correctness (brittle API contract)

**Problem:**
The winner highlight depends entirely on `player.result` being the exact string `'win'`. If the backend returns `'winner'`, `'WIN'`, `'won'`, or any other variant, all winner highlighting silently fails — no error is thrown, no warning is logged. There is no comment citing the API contract or a constants file that documents the expected string.

**Code:**
```js
const isWinner = player.result === 'win';
```

**Suggested Fix:**
Add an inline comment citing the API contract, or import a shared constant:
```js
// player.result: 'win' | 'loss' | 'fold' per /api/hands response schema
const isWinner = player.result === 'win';
```

**Impact:** Silent visual breakage if the server-side result vocabulary ever changes.

### [LOW] Winner background `#b8860b` (Dark Goldenrod) may read as dark amber rather than gold

**File:** `frontend/src/components/resultOverlay.js`
**Line(s):** 17
**Category:** convention (visual design intent)

**Problem:**
AC-3 requires a "gold background" for the winner row. The chosen colour, `#b8860b` (CSS Dark Goldenrod), renders as a notably dark amber-brown rather than the canonical gold (`#FFD700`). Against the overlay's dark background, the row is distinguishable, but on a standard monitor the hue is closer to bronze/dark amber than gold.

**Code:**
```js
row.style.cssText = 'background:#b8860b;font-weight:bold;';
```

**Suggested Fix:**
Use `#c9a227` (medium gold) or `#d4a017` for a brighter but still readable gold tone, or add `color:#000` to ensure sufficient contrast with any gold shade:
```js
row.style.cssText = 'background:#c9a227;color:#000;font-weight:bold;';
```

**Impact:** Subjective visual fidelity. The AC is technically met (gold-family colour, bold type) but may surprise a designer or product owner reviewing the UI.

### [LOW] All presentation styles are inlined; no CSS class usage

**File:** `frontend/src/components/resultOverlay.js`
**Line(s):** 17, 22–23, 29, 37, 47, 54, 68–78, 94–99, 103–108, 119, 125, 127, 133
**Category:** design (maintainability)

**Problem:**
Every visual aspect — layout, typography, colour, spacing — is encoded as `style.cssText` or `style.color` directly in JS. There is no use of CSS classes or a linked stylesheet. This makes theming, dark/light mode toggling, and style auditing significantly harder, as the "source of truth" for the overlay's appearance is scattered across 15+ `style` assignments spread through the function.

**Suggested Fix:**
Extract a `result-overlay.css` file (or a `<style>` tag injected once) and apply class names such as `.result-overlay-header`, `.result-overlay-winner`, `.result-overlay-card-red`. The overlay already carries `className = 'result-overlay'`, establishing the pattern.

**Impact:** No runtime bug. Increases friction for future style changes and violates the separation-of-concerns principle for frontend modules.
